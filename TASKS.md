# 프로젝트 전체 Task List (백로그 - 애자일 팀 구성용)

팀장님을 포함한 **4명의 개발팀**이 병렬로 작업을 나누어 진행할 수 있도록, 기존 Task에서 비어있던 '데이터 수집 공통화' 모듈 등을 추가하여 프로젝트의 무결점을 보장하는 형태로 분할하였습니다.

---

## 1. LangGraph Pipeline Skeleton 구축 (주춧돌)
### User Story
**As a** 시스템 아키텍트 (구조 담당),
**I want** LangGraph의 노드(Agent)와 분기(조건부 엣지)를 연결하는 전체 파이프라인(State Graph)을 빈 껍데기 형태로 구축하여,
**So that** 팀원들이 각자 맡은 모듈만 완성해서 끼워 넣으면 톱니바퀴처럼 전체 시스템이 동작하게 뼈대를 잡고 싶다.

### Acceptance Criteria
- [ ] 에이전트 간 주고받을 데이터 상태 변수 `State` 클래스(Pydantic 또는 TypedDict) 스키마 정의
- [ ] 각 전문 노드 간 결과물이 올바른 순서대로 넘어가는 더미(Dummy) 워크플로우 통과
- [ ] 시스템 아키텍처의 전체 그래프 실행 흐름 시각화 및 검증
- [ ] 에이전트 개발자들이 즉시 병렬 작업을 시작할 수 있도록, 각 노드에 전달될 하드코딩된 더미(Mock) 데이터 셋을 생성하여 공유
- [ ] **각 에이전트가 서로의 컨텍스트(프롬프트, 중간 사고 과정)에 오염되지 않도록, State 내에 에이전트별 독립된 메모리(격리된 상태값) 구조 설계**

---

## 2. Agent: 매크로 분석 (거시 시황 전문가) 구현
### User Story
**As a** 거시 분석 에이전트 담당 개발자,
**I want** 에이전트가 스스로 FRED나 뉴스 API 등을 호출하여 경제 지표 데이터를 확보한 뒤, 이를 바탕으로 LLM에게 프롬프트를 주입하여,
**So that** 리포트를 읽는 투자자가 글 서두에 "현재 시장 전체의 판이 어떻게 돌아가는지"를 2~3줄로 직관적으로 이해하게 하고 싶다.

### Acceptance Criteria
- [ ] 독자적으로 기준 금리와 최근 매크로 뉴스를 수집하는 로직(또는 Tool 결합) 구현
- [ ] 수집된 데이터를 바탕으로 출력된 텍스트가 사실적이고, 시장 상황을 깔끔하게 2-3줄의 시황으로 요약하는가?
- [ ] `State` 딕셔너리에 시황 분석 파트(Text) 정상 저장

---

## 3. Agent: Risk Alert (리스크 경고 전문가) 구현
### User Story
**As a** 리스크 관리 에이전트 담당 개발자,
**I want** 공매도 자금 이탈이나 하락 악재가 심한 섹터를 찾아내어 빨간불(경고)을 커주는 행동을 자동화하여,
**So that** 투자자가 포모(FOMO)에 빠져 상업용 오피스, 적자주 같은 '함정 주식'을 사는 것을 원천 봉쇄하고 싶다.

### Acceptance Criteria
- [ ] 기피해야 할 섹터의 논리적 배경(예: 고금리 롤오버 등)을 데이터 기반으로 LLM이 추론해 내는가?
- [ ] 절대 피해야 할 대표주 2~3개가 티커(Ticker)와 함께 강한 어조로 경고되는가?
- [ ] `State` 딕셔너리에 Risk 경고 파트(Text) 정상 저장

---

## 4. Agent: 알파 섹터 (주도주 추천) 구현
### User Story
**As a** 성장주/알파 에이전트 담당 개발자,
**I want** 데이터 기반으로 구조적 성장이 확실한 주도 섹터 및 양국 대장주를 선별하는 프롬프트를 구성하여,
**So that** 투자자가 포트폴리오를 주도주(예: 전력 기기, K-뷰티 수출주) 위주로 재편하여 초과 수익(Alpha)을 달성할 수 있게 하고 싶다.

### Acceptance Criteria
- [ ] 추천 섹터에 대한 논리적 배경과 명확한 투자 포인트가 제시되는가?
- [ ] 실질적 행동 매개체인 대상 타겟 종목들(이튼, 실리콘투 등)이 명시적으로 제시되는가?
- [ ] `State` 딕셔너리에 추천 파트(Text) 정상 저장

---

## 5. Agent: 개인 포트폴리오 진단 구현
### User Story
**As a** 프라이빗 뱅커(PB) 에이전트 담당 개발자,
**I want** 유저의 문제성 종목(예: 테슬라위클리커버드콜 등) 정보를 받아 현재 구조적인 한계를 진단하고 교체 매매를 제안하여,
**So that** 사용자(개인)가 나만의 맞춤형 계좌 멘토링을 받고 있다는 강렬한 구독 서비스를 체험하게 하고 싶다.

### Acceptance Criteria
- [ ] 주어진 종목(ETF 등)의 구조적 속성을 분석하고 현 매크로 상황과 대조를 수행하는가?
- [ ] '단순 매수/매도'가 아니라, 현재 종목의 리스크를 논리적으로 설명하고 구체적인 대안 상품(스위칭) 액션 플랜을 제시하는가?
- [ ] `State` 딕셔너리에 맞춤 진단 파트(Text) 정상 저장

---

## 6. Agent: GP (검토자 Feedback) 구현
### User Story
**As a** 품질 검수(Critic) 에이전트 개발자,
**I want** 거시, 리스크, 알파, 포트폴리오 에이전트가 단독으로 만든 결과물들을 가로채어 모순점을 체크하고, 오류가 있다면 "어느 에이전트"에게 "무엇을 수정하라"는 명확한 구조화된 피드백 객체(Feedback Data)를 생성하여,
**So that** 파이프라인(1번 담당자)이 이 객체를 보고 해당 에이전트 노드로 정확히 되돌려보내고(Routing), 각 에이전트가 스스로 피드백을 반영해 다시 글을 써오게(Re-run) 퀄리티를 통제하고 싶다.

### Acceptance Criteria
- [ ] 단순히 "다시 써(Fail)"라는 텍스트가 아니라, `{ "target_node": "risk_agent", "feedback_reason": "금리와 테슬라 주가의 연관성 논리가 부족함" }` 형태의 명확한 **라우팅용 구조화 데이터명세(Pydantic/JSON)** 를 반환하는가?
- [ ] 위에서 정의한 피드백 규격을 1번(파이프라인 설계자)과 2~5번(각 에이전트 담당자)에게 공유하여 연동(Integration)을 주도했는가? (즉, 각 에이전트 프롬프트에 'GP의 피드백이 들어오면 무시하지 말고 반영해서 다시 쓸 것'이라는 로직이 추가되도록 안내)
- [ ] 무한 루프로 재작업에 빠지지 않도록, State 내에 최대 재시도(Max Retry) 횟수를 카운팅하고 통제하는 로직이 있는가?

## 7. Agent: CIO (총괄 편집자) 구현
### User Story
**As a** 편집장 에이전트 담당 개발자,
**I want** GP의 검사를 무사히 통과한 모든 리포트 조각(State)을 끌어모아 통합하여,
**So that** 사용자가 여러 전문가의 의견이 아닌, 최고의 전문 애널리스트 한 명이 통일된 어투로 유려하게 써 내려간 글처럼 리포트를 편하게 읽게 하고 싶다.

### Acceptance Criteria
- [ ] 앞선 4개의 파트를 모아 지정된 Notion 마크다운 양식(제안서 참조) 템플릿으로 구조화
- [ ] 여러 에이전트의 말이 섞여서 발생할 수 있는 중복 내용을 제거하고 톤앤매너 통일
- [ ] 최종 문자열을 리턴

---

## 8. 외부 연동: Notion 퍼블리시
### User Story
**As a** 리포트 퍼블리셔(백엔드) 개발자,
**I want** CIO 에이전트가 완성시킨 리포트 텍스트를 Notion API 환경으로 다이렉트 푸시하여,
**So that** 마크다운(Heading, Bullet, Emoji) 디자인이 깨지지 않은 상태로 Notion 페이지에 최종 발행물로 등록하고 싶다.

### Acceptance Criteria
- [ ] Notion API Token 및 Database 통신 기반 구축
- [ ] 일자 정보(예: `[2026-03-24] 리포트 제목`)가 페이지 타이틀로 동적 생성
- [ ] 에이전트의 텍스트가 런타임에 성공적으로 Notion Page의 Block 단위로 전환되어 적재

## 9. 정량 평가 (Quantitative Evaluation)
### User Story
**As a** 성능 평가자 (LLMOps 담당),
**I want** RAGAS 같은 평가 프레임워크나 벤치마크 툴을 통해 에이전트 출력물의 정확도를 수치화하여,
**So that** 주가가 폭락하는데 매수를 추천하는 등의 대형 사고(환각) 확률이 몇 %인지 시스템적으로 모니터링하고 싶다.

### Acceptance Criteria
- [ ] 답변 릴러번시(Answer Relevancy)나 맥락 팩트 기반성(Faithfulness) 등 최소 2개 이상의 매트릭 자동 채점
- [ ] 오픈소스 기반 파이프라인(로컬 평가 구동) 구축 여부

---

## 10. 정성 평가 (Qualitative Evaluation)
### User Story
**As a** 프로덕트 매니저(PM/PO),
**I want** 사람(개발자)이 매번 눈으로 읽으며 감탄하지 않아도, 뛰어난 LLM(예: GPT-5)을 심사관(LLM-as-a-judge)으로 설정하여 글의 매력도를 테스트하고,
**So that** "이 리포트를 기꺼이 돈을 내고 살 의향이 있는가?"라는 유료화 수준(PMF) 충족 여부를 별점 5점 만점으로 추적하고 싶다.

### Acceptance Criteria
- [ ] 평가용 심사관 프롬프트 (재치, 설득력, 논리성 기준표) 디자인
- [ ] 무작위로 생성한 3~5개의 테스트 런 결과에 대해 평균 평점을 산출하는 스크립트 도출

## 11. (Optional) 공통 API 도구함 통합 및 리팩토링
### User Story
**As a** 데이터 엔지니어 (또는 공통 모듈 담당자),
**I want** 각 에이전트 개발자들이 각자 알아서 (개별적으로) 작성한 데이터 수집 코드나 프롬프트를 나중에 분석하여, 중복으로 호출되는 API(예: 환율, 동일 지수 등)를 하나의 공통 모듈 혹은 라이브러리(Tool)로 묶어내어,
**So that** 초기발진(MVP) 단계에서는 에이전트 개발자들이 남을 기다리지 않고 마음껏 자체 데이터를 당겨 쓰게 하되, 추후 프로젝트 최적화 시점(Phase 2)에 발생할 수 있는 API 비용 낭비와 유지보수 불균형을 해결하고 싶다.

### Acceptance Criteria
- [ ] (MVP 이후 진행) 각 에이전트(거시, 리스크 등) 내부에 산재해 있는 API 호출 로직(yfinance, FRED 등) 파악
- [ ] 중복 검색되는 핵심 지표를 하나로 일원화하는 공통 함수(Utility) 작성 및 기존 에이전트 코드 리팩토링
- [ ] (선택) 에이전트들이 공통으로 쓸 수 있는 통합된 `Tool` 객체 세트 제공
